# Keycloak household identity operations

The realm import creates no users and contains no real credentials. Keycloak is a browser-facing identity endpoint: configure `KEYCLOAK_PUBLIC_URL` and `OIDC_ISSUER_URI` to the same exact LAN/VPN-reachable origin, and restrict that listener with the host firewall or VPN policy. The gateway resolves that public hostname back to the host so discovery and JWT `iss` validation use the same issuer. The committed realm permits only the origins `https://localhost:4200`, `https://192.168.0.100:4200`, and `http://localhost:4200`; replace these exact entries when the supported frontend origin changes. Start Keycloak with local values supplied in `.env`, sign in to the admin console, create the first household administrator, assign `household-admin`, and immediately rotate the bootstrap administrator password.

Login is password-based with long remember-me sessions (idle 30d / max 90d); users should store the password in a device password manager so re-login is an autofill. Passkeys (WebAuthn) were implemented and then abandoned: the RP ID requires a DNS name, which forced a LAN DNS dependency the household rejected. Recovery is performed by a household administrator from the loopback admin console (or `kcadm.sh set-password`). Self-registration remains disabled. Note the realm JSON only imports into an empty database — apply changes to a live realm with `kcadm.sh` and mirror them in the JSON.

Every agent must have its own confidential client. Store its secret outside the repository, grant only required scopes, and disable that client to revoke it. Do not share the smoke-test client with a production agent.
